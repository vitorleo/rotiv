## Summary

## What Changed (and What Worked)

### ✓ Worker is now extracted from the binary
The `E_WORKER_NOT_FOUND` error from v0.1.0 is gone. The binary now correctly embeds and extracts
the 6 route-worker TypeScript source files to a temp directory at runtime.

### ✓ Migrate now resolves from the project dir
`rotiv migrate` no longer says "run from the Rotiv monorepo". It now resolves correctly from the
project directory (though it still fails — see below).

### ✓ CLI tooling remains excellent
`rotiv new`, `rotiv add model`, `rotiv add route`, `rotiv spec-sync`, `rotiv validate`,
`rotiv context-regen`, `rotiv diff-impact`, and `rotiv explain` all work perfectly.
`rotiv validate` passed with zero issues on the completed blog app.

### ✓ E011 now suggests the correct name
`rotiv add model post` now returns a corrected suggestion (`Post`) in the error output. Good fix.

### ✓ Dev banner shows full file paths
`app/routes/posts/new.tsx` and `app/routes/posts/[slug].tsx` now display correctly in the startup
banner. This regression from v0.1.0 is fixed.

## Remaining Blockers

### Bug 1 (New in v0.2.0): Worker temp dir missing `"type": "module"`

**Steps to reproduce:**
```bash
rotiv new blog
cd blog
# manually install tsx (see Bug 2)
rotiv dev
```

**Error:**
```
Error: Transform failed with 1 error:
.../src/index.ts:68:0: ERROR: Top-level await is currently not supported with the "cjs" output format
```

**Root cause:** The binary extracts 6 `.ts` files to a temp directory but does not write a
`package.json` alongside them. Without `"type": "module"`, Node.js and `tsx`/`esbuild` treat the
directory as CommonJS. The worker uses top-level `await` (line 68 of `index.ts`) which is only
valid in ESM.

**Fix (one line):** When extracting worker files, write `{"type":"module"}` as
`$TMPDIR/package.json`. This is the only change needed.

---

### Bug 2 (Persists from v0.1.0): `@rotiv/*` npm packages still not published

`pnpm install` (or `npm install`) still fails with 404 for all `@rotiv/*` packages.
The v0.2.0 changelog says they were published, but they are not on the npm registry as of 2026-03-15.

As a workaround I installed `tsx` and `drizzle-kit` by temporarily swapping `package.json`
for a stub. This is not a reasonable expectation for end users.

**Impact:** Without `tsx` in `node_modules`, both `rotiv dev` and `rotiv migrate` fail immediately.

---

### Bug 3 (Persists): `rotiv migrate` error message swallows stderr

When `drizzle-kit` fails, the error is:
```json
{"error": "Error: drizzle-kit migrate failed:\n"}
```
The actual drizzle-kit error output (missing config, missing deps, etc.) is not captured or
forwarded. Agents — and humans — have no way to diagnose what went wrong.

**Fix:** Capture both stdout and stderr from the `drizzle-kit` child process and include them in
the structured error output.

## One-Line Fix for Bug 1

In `crates/rotiv-cli/src/commands/dev.rs`, after writing the worker source files to `$tmpdir`:

```rust
fs::write(tmpdir.path().join("package.json"), r#"{"type":"module"}"#)?;
```

This is the minimum change to make `rotiv dev` functional on a standalone install.
