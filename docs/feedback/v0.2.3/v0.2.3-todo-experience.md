## What v0.2.3 Tried to Fix (and Why It Didn't)

The changelog states that `resolve_tsx_loader()` now searches these paths (in order):

1. `<project_dir>/node_modules/tsx/dist/esm/index.cjs`
2. `<binary_dir>/node_modules/tsx/dist/esm/index.cjs`
3. `<binary_dir>/../node_modules/tsx/dist/esm/index.cjs`
4. Falls back to bare `"tsx"` (globally installed)

**What's actually on this machine:**

| Path searched | Exists? |
|---|---|
| `todo/node_modules/tsx/...` | ❌ (no node_modules — pnpm install failed) |
| `bin/node_modules/tsx/...` | ❌ (no node_modules next to binary) |
| `bin/../node_modules/tsx/...` | ❌ |
| Global tsx | ✓ at `C:\Users\Vitor\AppData\Roaming\npm\node_modules\tsx` |

All three explicit paths miss. The fallback is `"tsx"` (bare name), which is passed to
`node --import tsx` — but Node.js resolves `--import tsx` using the same module resolution
as any other package import, starting from the project dir. Since the project dir has no
`node_modules/tsx`, it fails.

## Root Cause Analysis

The v0.2.3 fix has two problems:

### Problem 1: Global npm path not searched

On Windows, the global npm package dir is:
`C:\Users\<user>\AppData\Roaming\npm\node_modules\`

This is not in any of the 3 paths searched by `resolve_tsx_loader()`. The fix would need to
also run `npm root -g` (or check `%APPDATA%\npm\node_modules\tsx`) to find globally installed packages.

### Problem 2: Bare `"tsx"` fallback still uses Node module resolution

When all path checks fail, the binary falls back to passing `"tsx"` to `node --import tsx`.
But `--import` uses Node's standard ESM resolution, which searches `node_modules` up the
directory tree from the project — not PATH, not the global npm dir.

The correct fallback for a globally installed package is to pass the **absolute path** from
`which tsx` or `npm root -g`. For example:

```rust
// Run: npm root -g → "C:\Users\Vitor\AppData\Roaming\npm\node_modules"
// Then check: <npm_root>/tsx/dist/esm/index.cjs
let npm_root = std::process::Command::new("npm").args(["root", "-g"]).output()...;
```

Or even simpler: if `tsx` is on PATH, use `where tsx` (Windows) / `which tsx` (Unix) to get
its bin path, then derive the package path from it.

## Suggested Fix for v0.2.4

```rust
fn resolve_tsx_loader(project_dir: &Path) -> String {
    let candidates = vec![
        // 1. Project node_modules (works when pnpm install succeeds)
        project_dir.join("node_modules/tsx/dist/esm/index.cjs"),
        // 2. Binary-adjacent node_modules
        binary_dir().join("node_modules/tsx/dist/esm/index.cjs"),
        // 3. Global npm root (cross-platform)
        npm_global_root().join("tsx/dist/esm/index.cjs"),
    ];

    for path in &candidates {
        if path.exists() {
            return path.to_string_lossy().into_owned();
        }
    }

    // Last resort: resolve via `npm root -g` at runtime
    if let Ok(output) = Command::new("npm").args(["root", "-g"]).output() {
        let npm_root = String::from_utf8_lossy(&output.stdout).trim().to_string();
        let tsx_path = PathBuf::from(&npm_root).join("tsx/dist/esm/index.cjs");
        if tsx_path.exists() {
            return tsx_path.to_string_lossy().into_owned();
        }
    }

    "tsx".to_string() // bare fallback — will only work if in project node_modules
}
```
